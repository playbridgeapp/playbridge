# Mutable Playlist Feature ‚Äî Implementation Plan
_Last updated: 2026-04-01_

## Overview

The goal is to make the TV maintain a mutable, live playlist that the phone can add to, remove from, reorder, and jump around in ‚Äî without interrupting the currently playing video. The infrastructure is roughly 60% done; the gaps are in the phone UI, phone consuming TV feedback, missing mutation commands, and some TV-side coordination.

The unifying design decision: **every video send from the phone becomes a playlist**, even a single item. `Command.Play` is retired for internal players. `Command.Playlist` with `replace = true` is "play now and clear queue"; `replace = false` (or `Command.QueueAdd`) is "append without interrupting."

The TV maintains a single mutable playlist per session and broadcasts its state back to the phone after every mutation, so the phone's queue UI stays in sync.

---

## What's Already Done

### Protocol (`protocol/src/main/java/com/playbridge/protocol/Message.kt`)
- `Command.Playlist` + `PlaylistPayload(items, startIndex)` ‚Äî sends full playlist, replaces queue
- `Command.QueueAdd` + `QueueAddPayload(item)` ‚Äî append single item to active queue
- `Command.PlaylistJump` + `PlaylistJumpPayload(index)` ‚Äî jump to index
- `PlaylistStatusMessage(items, currentIndex, totalCount)` ‚Äî TV ‚Üí Phone broadcast
- All helper functions: `createPlaylistCommandJson`, `createQueueAddCommandJson`, `createPlaylistJumpCommandJson`, `createPlaylistStatusJson`

### TV ‚Äî `ServerService.kt`
- Handles `Command.Playlist` ‚Äî clears `pendingQueueItems`, launches player activity
- Handles `Command.QueueAdd` ‚Äî buffers in `pendingQueueItems`, broadcasts `ACTION_QUEUE_ADD`
- Handles `Command.PlaylistJump` ‚Äî broadcasts `ACTION_PLAYLIST_JUMP` with index
- `drainPendingQueueItems()` ‚Äî player drains this on start and on each `ACTION_QUEUE_ADD`

### TV ‚Äî `ExoPlayerActivity.kt`
- `playlistItems: MutableList<PlayPayload>` + `playlistIndex: Int`
- Receives `ACTION_QUEUE_ADD` ‚Üí appends items + calls `broadcastPlaylistStatus()`
- Receives `ACTION_PLAYLIST_JUMP` ‚Üí calls `playItemAtIndex(index)`
- `playNextInPlaylist()` ‚Äî auto-advances with progress save before each transition
- `playItemAtIndex(index)` ‚Äî saves progress then jumps, calls `broadcastPlaylistStatus()`
- `broadcastPlaylistStatus()` ‚Äî serialises playlist state and sends to `ServerService`

### TV ‚Äî `HistoryStore.kt` + `PlaybackHistoryItem`
- Already has `playlistJson: String?` and `playlistIndex: Int` fields on history items
- Uses DataStore (not Room), max 50 items, thumbnails captured via PixelCopy

## What Needs to Be Built

### Step 1 ‚Äî Unify "Play Now" into the playlist path
**Status: Not started**

**Protocol (`Message.kt`)**
- Add `replace: Boolean = true` to `PlaylistPayload` ‚Äî `true` = clear queue and start playing, `false` = append
- Add `sessionId: String?` to `PlaylistPayload` ‚Äî UUID generated by TV at the start of each new replace session; used as stable history entry ID as queue grows mid-session

**Phone ‚Äî `BrowserActivity.kt`**
- Replace `createPlayCommandJson(...)` with `createPlaylistCommandJson(listOf(playPayload), replace = true)` in the single-video send path (the `onVideoClick` handler)
- `Command.Play` can be kept in the protocol for backwards compat with older TV versions but is no longer sent by the phone for internal players

**TV ‚Äî `ServerService.kt`**
- `Command.Playlist` handler: check `replace` flag
  - `replace = true` (default): generate new UUID session ID, clear `pendingQueueItems`, launch player activity as now
  - `replace = false`: append all items to `pendingQueueItems` and broadcast `ACTION_QUEUE_ADD` (no new activity launch)
- Keep `Command.Play` handler intact for external player paths (`external`, `external_mpv`) ‚Äî those bypass ExoPlayerActivity entirely so playlist logic doesn't apply

**TV ‚Äî `ExoPlayerActivity.kt`**
- Accept and store `sessionId` from the playlist intent extra
- Pass `sessionId` through to `ProgressManager` so history entries use it as their `id`

---

### Step 2 ‚Äî "Add to Queue" button in phone UI
**Status: Not started**

**Phone ‚Äî `CastSheet` (wherever the "Play on TV" button lives)**
- Add a secondary "Add to Queue" button alongside "Play on TV"
- Only enabled when `connectionState == Connected` AND `tvActiveContext == "player"` ‚Äî requires an active player session to queue into
- On tap: send `createQueueAddCommandJson(playPayload)` ‚Äî **no** tab media pause (user is queuing, not replacing)
- On tap: do NOT call `tabManager.pauseMedia(session)` ‚Äî only "Play Now" pauses the tab

---

### Step 3 ‚Äî Phone receives and displays TV playlist state
**Status: Not started**

**Phone ‚Äî `ConnectionViewModel.kt`**
- Parse incoming `playlist_status` messages from the TV WebSocket (currently only `status` and `context` types are handled)
- Expose `val playlistStatus: StateFlow<PlaylistStatusMessage?>` ‚Äî resets to `null` when TV goes idle

**Phone ‚Äî `RemoteControlScreen.kt`**
- Add a collapsible queue strip below existing playback controls, driven by `playlistStatus`
- Show all queued item titles, highlight the current index
- Tap any item ‚Üí send `createPlaylistJumpCommandJson(index)`
- Show total count + current position ("3 / 12")
- Strip should be hidden / collapsed when `playlistStatus` is null or has ‚â§ 1 item

---

### Step 4 ‚Äî Missing mutation commands
**Status: Not started**

**Protocol (`Message.kt`)** ‚Äî add for each:
- `playlist_remove` ‚Äî `PlaylistRemovePayload(index: Int)`, `Command.PlaylistRemove`, parser case, `createPlaylistRemoveCommandJson(index)`
- `playlist_move` ‚Äî `PlaylistMovePayload(from: Int, to: Int)`, `Command.PlaylistMove`, parser case, `createPlaylistMoveCommandJson(from, to)`
- `playlist_clear` ‚Äî no payload, `Command.PlaylistClear`, parser case, `createPlaylistClearCommandJson()`

**TV ‚Äî `ServerService.kt`**
- Handle each new command by broadcasting a corresponding new action constant to the player activity

**TV ‚Äî `ExoPlayerActivity.kt`**
- `removeItemAtIndex(index)` ‚Äî remove item, adjust `playlistIndex` if needed (if removed index < current, decrement; if removed index == current, play next), call `broadcastPlaylistStatus()`
- `moveItem(from, to)` ‚Äî reorder list, adjust `playlistIndex` to follow the currently playing item, call `broadcastPlaylistStatus()`
- `clearQueue()` ‚Äî clear `playlistItems`, let current item finish, call `broadcastPlaylistStatus()`

**Phone ‚Äî `RemoteControlScreen.kt`**
- Swipe-to-remove on queue strip items ‚Üí send `createPlaylistRemoveCommandJson(index)`
- Drag-to-reorder ‚Üí send `createPlaylistMoveCommandJson(from, to)`
- "Clear queue" button ‚Üí send `createPlaylistClearCommandJson()`

---

### Step 5 ‚Äî VlcPlayerActivity and MpvPlayerActivity parity
**Status: Not started**

Both currently ignore all queue-related broadcasts. They need:
- Same `playlistItems` + `playlistIndex` state fields
- Register `ACTION_QUEUE_ADD`, `ACTION_PLAYLIST_JUMP`, and the three new action receivers from Step 4
- Implement `playNextInPlaylist()` and `playItemAtIndex()` (or extract to a shared base class / helper)
- Call `broadcastPlaylistStatus()` after every mutation

> **Note:** Consider extracting shared playlist logic into `PlayerActivity.kt` (the existing abstract base) so it isn't duplicated across all three player activities.

---

### Step 6 ‚Äî History: playlist entries
**Status: Not started**

**TV ‚Äî `HistoryStore.kt` / `PlaybackHistoryItem`**
- For multi-item playlists: use the session UUID (from Step 1) as the history `id` instead of the URL ‚Äî keeps the entry stable as items are queued mid-session and `saveProgress()` overwrites it with the latest `playlistJson`
- Single-item playlists continue using the URL as `id` (no change needed)

**TV ‚Äî `LibraryScreen.kt`**
- Detect playlist entries by `playlistJson != null && items.size > 1`
- Show episode count badge on the card ("Ep 3 / 12") and a small playlist/stack icon
- Progress bar reflects position within the current episode (not overall playlist completion)
- Tapping a playlist history entry: deserialise `playlistJson` ‚Üí inject into `PlaylistStore` ‚Üí launch player with `startIndex = playlistIndex` ‚Üí restore position within that episode

---

### Step 7 ‚Äî Cleanup
**Status: Not started**

**TV ‚Äî `ExoPlayerActivity.kt`**
- `broadcastPlaylistStatus()` currently builds JSON manually with `org.json.JSONObject` ‚Äî switch to `createPlaylistStatusJson()` from the protocol module for consistency

## Execution Order

```
Step 1 (unify send path) ‚Üí Step 2 (queue button) ‚Üí Step 3 (phone queue UI)
       ‚Üì
Step 4 (mutation commands) ‚Üí Step 5 (VLC/MPV parity) ‚Üí Step 6 (history) ‚Üí Step 7 (cleanup)
```

Steps 1‚Äì3 form the core usable feature (play now, queue next, see queue on phone). Steps 4‚Äì7 are the fuller experience and can be done in a second pass.
